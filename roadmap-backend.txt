1. VISÃO, STACK E CONVENÇÕES DA API
================================================================================

1.1 Stack sugerida
- Runtime: Python 3.12+
- Framework HTTP: FastAPI
- ORM: SQLAlchemy 2.x ou equivalente
- Validação: Pydantic v2
- Auth: JWT (access + refresh), hash de senha (argon2/bcrypt)
- DB: PostgreSQL 16+
- Objeto/Blob: S3-compatível ou Azure Blob (URLs assinadas)
- Filas/worker: Celery + Redis, RQ, ou arq (jobs IA e correção)
- Opcional: WebSocket ou SSE para chat do assistente (baixa latência)

1.2 Multi-tenant
- Raiz lógica: instituicao_id (UUID) em todas as entidades de negócio.
- Toda query de leitura/escrita deve filtrar por instituicao_id derivado do
  usuário autenticado (exceto tabelas globais de sistema, se existirem).
- Não retornar dados de outra instituição mesmo com ID válido (teste de IDOR).

1.3 Prefixo e versão
- Base URL: /api/v1
- Exemplo: POST /api/v1/auth/login

1.4 Autenticação
- Header: Authorization: Bearer <access_token>
- Access token: curta duração (ex.: 15–30 min).
- Refresh token: rotação em cookie HttpOnly ou body (definir política CSRF se cookie).

1.5 Identificadores e tempo
- IDs públicos: UUID string no JSON (ex.: "9f422730-261f-4196-8f2a-e807d5f6bb20").
- Timestamps no banco: timestamptz, sempre armazenados em UTC.
- Campos sugeridos: criado_em, atualizado_em, publicado_em, encerrada_em, lido_em.

1.6 Formato de erro (envelope JSON)
{
  "code": "STRING_MACHINE_READABLE",
  "message": "Mensagem legível",
  "details": { "campo": ["erro de validação"], ... }   // opcional
}
HTTP: usar 400 (validação), 401 (não autenticado), 403 (proibido), 404, 409
(conflito, ex.: unicidade submissão), 422 (FastAPI validation), 500.

1.7 Paginação e cursores
- Listas longas: cursor opcional (opaque) ou page + page_size (max 100).
- Resposta paginada:
{
  "items": [ ... ],
  "next_cursor": "string|null",
  "has_more": boolean
}

1.8 CORS e upload
- CORS restrito aos domínios do frontend em produção.
- Upload direto ao blob storage via presign; backend valida MIME, tamanho, tenant.

================================================================================
2. PERFIS RBAC E MATRIZ DE AUTORIZAÇÃO
================================================================================

2.1 Perfis (tipo_perfil em usuario_conta)
- professor: administrador pedagógico no escopo das turmas/instituição vinculadas.
- aluno: consome materiais autorizados; responde avaliações publicadas no prazo.
- responsavel: leitura de dados dos dependentes (dashboard, relatórios, comunicados).

2.2 Regras gerais (RF-002)
- Aluno e responsável: sem mutação de matérias, questões, comunicados institucionais,
  publicação de avaliações, etc.
- Exceções de escrita para aluno: criar/atualizar/enviar PRÓPRIA submissão; marcar leitura
  de comunicado quando aplicável.
- Professor: CRUD pedagógico completo no seu escopo.

2.3 Matriz resumida (endpoint → perfis)
Legenda: P=professor, A=aluno, R=responsavel

/auth/login, /auth/refresh, /auth/logout, /auth/me, /users/me/preferences → P,A,R
/instituicoes/{id} → P,A,R (somente leitura do próprio contexto)
/turmas*, /alunos*, matrículas (leitura para montar destinatários) → P (gestão); A/R conforme escopo
/search → P (federado docente); A/R versão restrita (opcional MVP: só P)
/notificacoes* → P,A,R (cada um vê as suas)
/avaliacoes/materias*, assuntos*, pastas*, avaliacoes*, questoes*, chat* → P
/aluno/avaliacoes*, /aluno/submissoes* → A
/conteudo/* (CRUD materiais/pastas) → P; GET materiais autorizados → A (e R se política permitir leitura)
/comunicados* (criar/editar/publicar) → P; GET inbox → P,A,R escopado
/dashboard/resumo → P; R (visão tutelados); A omitido ou futuro “meu desempenho”
/submissoes/{id}/relatorio-ia → P, R (dependente), A (próprio)

3. SERVIÇOS (ENDPOINTS REST) — ESPECIFICAÇÃO COMPLETA
================================================================================

Convenção: todos os paths abaixo são relativos a /api/v1

--------------------------------------------------------------------------------
3.1 AUTENTICAÇÃO E SESSÃO
--------------------------------------------------------------------------------

POST /auth/login
  Perfis: público
  Body (LoginRequest):
    email: str (formato email, obrigatório)
    senha: str (obrigatório)
  Response 200 (LoginResponse):
    access_token: str
    refresh_token: str
    token_type: "bearer"
    expira_em: str (ISO 8601 UTC do access)
    usuario: UserMe
  Erros: 401 credenciais inválidas; 403 conta desativada
  RN: incrementar contador de falhas; bloqueio opcional após N tentativas.

POST /auth/refresh
  Body:
    refresh_token: str
  Response 200: mesmo contrato de tokens + UserMe resumido (opcional)
  Erros: 401 refresh inválido/expirado

POST /auth/logout
  Perfis: autenticado
  Body opcional: refresh_token (para revogar)
  Response 204
  RN: invalidar refresh (tabela blacklist ou revogar sessão).

GET /auth/me
  Perfis: autenticado
  Response 200: UserMe

PATCH /users/me/preferences
  Perfis: autenticado
  Body (UserPreferencesPatch):
    tema: "light"|"dark"|"system"|null (opcional)
    idioma: str|null (ex.: "pt-BR", opcional)
    densidade_ui: "comfortable"|"compact"|null (opcional, futuro)
  Response 200: UserMe (preferencias atualizadas)
  RN: merge parcial; não alterar email/senha aqui.

--------------------------------------------------------------------------------
3.2 INSTITUIÇÃO E CADASTROS BASE
--------------------------------------------------------------------------------

GET /instituicoes/{instituicao_id}
  Perfis: autenticado (somente se instituicao_id == do token)
  Response 200 (InstituicaoResponse):
    id, nome_fantasia, documento_legal (opcional), configuracoes (objeto opcional)

GET /turmas
  Query: cursor, nome (filtro opcional)
  Perfis: P (todas as turmas do escopo); A/R (turma(s) vinculadas)
  Response: Paginated<TurmaListItem>
    TurmaListItem: id, nome, ano_letivo, turno (opcional), professor_titular_nome (opcional)

GET /turmas/{turma_id}
  Perfis: escopo
  Response 200: TurmaDetail (TurmaListItem + contagem_alunos opcional)

GET /turmas/{turma_id}/alunos
  Perfis: P (da turma); R (se turma contém dependente)
  Response: lista de AlunoListItem: id, nome_exibicao, usuario_id (opcional)

GET /alunos/{aluno_id}
  Perfis: P; R (se dependente); A (se próprio)
  Response: AlunoDetail

GET /alunos/{aluno_id}/responsaveis
  Perfis: P; A (próprio)
  Response: lista ResponsavelListItem

Notas MVP:
- CRUD de turmas/alunos pode ser seed/admin fora do escopo UI atual; expor leitura mínima
  para destinatários de comunicados e dashboard.

--------------------------------------------------------------------------------
3.3 BUSCA FEDERADA (CABEÇALHO)
--------------------------------------------------------------------------------

GET /search
  Query:
    q: str (obrigatório, min 2 caracteres)
    types: str opcional, CSV: aluno,material,avaliacao,comunicado (default: todos permitidos ao perfil)
    limit: int (default 20, max 50)
  Perfis: P (completo no escopo); A/R versão restrita (opcional: só materiais + comunicados)
  Response 200 (SearchResponse):
    grupos: [
      { "tipo": "aluno", "itens": [ SearchHit, ... ] },
      ...
    ]
  SearchHit:
    id: str
    tipo: str
    titulo: str
    subtitulo: str|null
    url_deep_link: str (path frontend sugerido, ex.: /avaliacoes/{mat}/{pasta}/{av})
  RN: respeitar instituicao_id; não vazar rascunhos de avaliação para aluno; comunicados só
  se usuário é destinatário efetivo.

3.4 NOTIFICAÇÕES (CABEÇALHO)
--------------------------------------------------------------------------------

GET /notificacoes
  Query: lida: bool|null, cursor
  Perfis: autenticado
  Response: Paginated<NotificacaoItem>
  NotificacaoItem:
    id, titulo, corpo_curto, tipo_evento, lida: bool, criado_em, link_profundo (opcional)

PATCH /notificacoes/{notificacao_id}/lida
  Body: { "lida": true }
  Perfis: dono da notificação
  Response 200: NotificacaoItem

POST /notificacoes/marcar-todas-lidas
  Perfis: autenticado
  Response 200: { "atualizadas": int }

Eventos que geram Notificacao (exemplos):
- comunicado publicado destinado ao usuário
- relatorio_ia disponível para submissão do dependente (responsavel)
- avaliação publicada na turma do aluno (opcional)

--------------------------------------------------------------------------------
3.5 MÓDULO AVALIAÇÕES (HIERARQUIA: MATÉRIA → ASSUNTO → PASTA → AVALIAÇÃO → QUESTÃO)
--------------------------------------------------------------------------------

--- Matérias ---

GET /avaliacoes/materias
  Perfis: P
  Response 200: lista MateriaResponse
  MateriaResponse:
    id: UUID
    nome: str
    cor: str (token Tailwind ou código hex — compatível com UI)
    conteudos_count: int (derivado: número de pastas de avaliação ou conteúdos)
    assuntos_count: int (opcional)

POST /avaliacoes/materias
  Perfis: P
  Body (MateriaCreateRequest):
    nome: str
    cor: str|null (se null, servidor sorteia ou aplica padrão)
  Response 201: MateriaResponse
  RN: criar automaticamente um Assunto inicial com nome "Geral" e ordem 0
    (espelha avaliacoes-provedor.tsx).

PATCH /avaliacoes/materias/{materia_id}
  Body (MateriaPatchRequest): nome (opcional), cor (opcional), ordem (opcional int)
  Response 200: MateriaResponse

DELETE /avaliacoes/materias/{materia_id}
  Perfis: P
  Response 204 ou 409 se existir submissão em qualquer avaliação filha (política recomendada:
  bloquear exclusão se houver submissao; alternativa: soft-delete)

GET /avaliacoes/materias/{materia_id}/arvore
  Perfis: P
  Response 200 (MateriaArvoreResponse):
    materia: MateriaResponse
    assuntos: [
      {
        id, nome, ordem,
        pastas: [ PastaAvaliacoesResumo, ... ]
      }
    ]
  PastaAvaliacoesResumo:
    id, nome,
    alunos_responderam, alunos_total,
    avaliacoes_concluidas, avaliacoes_total,
    status_resumo: str (texto derivado para UI)

--- Assuntos ---

POST /avaliacoes/materias/{materia_id}/assuntos
  Body: { "nome": str, "ordem": int|null }
  Response 201: AssuntoResponse (id, materia_id, nome, ordem)

PATCH /avaliacoes/assuntos/{assunto_id}
  Body: nome (opcional), ordem (opcional)
  Response 200: AssuntoResponse

DELETE /avaliacoes/assuntos/{assunto_id}
  Response 204 ou 409 (mesma política de submissões)

--- Pastas de avaliações (equivalente ConteudoAvaliacao na UI) ---

POST /avaliacoes/assuntos/{assunto_id}/pastas
  Body (PastaAvaliacoesCreateRequest):
    nome: str|null — se vazio, default "Nova pasta de avaliações"
  Response 201: PastaAvaliacoesDetail
  PastaAvaliacoesDetail:
    id, assunto_id, nome,
    alunos_responderam, alunos_total,
    avaliacoes_concluidas, avaliacoes_total,
    status_resumo: str

GET /avaliacoes/pastas/{pasta_id}
  Perfis: P
  Response 200: PastaAvaliacoesDetail
  RN: campos agregados são SEMPRE derivados de submissões e estados das avaliações,
    nunca aceitos como input de escrita.

PATCH /avaliacoes/pastas/{pasta_id}
  Body: { "nome": str }
  Response 200: PastaAvaliacoesDetail

--- Avaliações ---

GET /avaliacoes/pastas/{pasta_id}/avaliacoes
  Perfis: P
  Response 200: lista AvaliacaoListItemResponse
  AvaliacaoListItemResponse (espelha AvaliacaoListaItem):
    id, titulo, status: "rascunho"|"publicada"|"encerrada",
    alunos_feitos: int, alunos_total: int

POST /avaliacoes/pastas/{pasta_id}/avaliacoes
  Body opcional: { "titulo": str|null } — default "Nova avaliação"
  Response 201: AvaliacaoDetailResponse (rascunho, sem questões ou vazio)

GET /avaliacoes/avaliacoes/{avaliacao_id}
  Perfis: P (sempre); A via rota /aluno/... (sem gabarito)
  Response 200: AvaliacaoDetailResponse
  AvaliacaoDetailResponse:
    id, pasta_id, titulo, status,
    publicado_em: datetime|null, encerrada_em: datetime|null,
    prazo_utc: datetime|null,
    questoes: [ QuestaoResponse, ... ],
    payload_editor_jsonb: object|null (snapshot do editor, opcional),
    pode_editar: bool (derivado: status == rascunho)

PATCH /avaliacoes/avaliacoes/{avaliacao_id}
  Perfis: P
  Body (AvaliacaoPatchRequest):
    titulo: str|null (opcional)
    prazo_utc: datetime|null (opcional)
  RN: não permitir mudança de status por este PATCH; usar ações dedicadas.
  Erros: 403 se status != rascunho para campos sensíveis (ou permitir só prazo em publicada
    conforme política; MVP recomendado: só rascunho edita título/prazo de edição)

POST /avaliacoes/avaliacoes/{avaliacao_id}/salvar-rascunho
  Perfis: P
  Body (AvaliacaoSalvarRascunhoRequest):
    titulo: str|null (opcional)
    prazo_utc: datetime|null (opcional)
    questoes: [ QuestaoUpsert, ... ]  // ver DTOs seção 4
    payload_editor: object|null (opcional, JSON serializável)
  Response 200: AvaliacaoDetailResponse
  RN: idempotente por versão opcional (If-Match / versao int); RF-007 auto-save debounce.
  Erros: 403 se não rascunho.

POST /avaliacoes/avaliacoes/{avaliacao_id}/publicar
  Perfis: P
  Body opcional: { "prazo_utc": datetime|null }
  RN:
    - status atual deve ser rascunho
    - deve haver pelo menos 1 questão válida
    - toda múltipla escolha deve ter alternativas >= 2 e resposta_correta no intervalo
  Response 200: AvaliacaoDetailResponse (status publicada, publicado_em preenchido)
  Erros: 409 se já publicada/encerrada

POST /avaliacoes/avaliacoes/{avaliacao_id}/encerrar
  Perfis: P
  RN: transição publicada → encerrada; irreversível MVP; rejeitar novas submissões/alterações
  Response 200: AvaliacaoDetailResponse
  Erros: 409 se rascunho ou já encerrada

--- Questões (alternativa granular além do salvar-rascunho bulk) ---

PUT /avaliacoes/avaliacoes/{avaliacao_id}/questoes
  Body: { "questoes": [ QuestaoUpsert, ... ] } — substitui ordem e conteúdo completo
  Perfis: P, apenas rascunho
  Response 200: lista QuestaoResponse

POST /avaliacoes/avaliacoes/{avaliacao_id}/questoes
  Body: QuestaoUpsert (sem id cria novo)
  Response 201: QuestaoResponse

PATCH /avaliacoes/avaliacoes/{avaliacao_id}/questoes/{questao_id}
  Body: parcial QuestaoUpsert
  Response 200: QuestaoResponse

DELETE /avaliacoes/avaliacoes/{avaliacao_id}/questoes/{questao_id}
  Response 204

POST /avaliacoes/avaliacoes/{avaliacao_id}/questoes/reordenar
  Body: { "ordem_ids": [ UUID, ... ] }
  Response 200: lista QuestaoResponse

--- Assistente IA (sidecar) ---

POST /avaliacoes/avaliacoes/{avaliacao_id}/chat
  Perfis: P
  Body (AvaliacaoChatMessageRequest):
    mensagem: str
    cliente_msg_id: str|null (idempotência opcional)
  Response 200 (sync) ou 202 (async):
    modo sync: { "resposta": str, "mensagem_id": UUID }
    modo async: { "job_id": UUID, "status": "queued" }
  RN: só professor; avaliação em rascunho recomendado para aplicar mudanças; log de auditoria
    opcional para compliance IA.

GET /avaliacoes/avaliacoes/{avaliacao_id}/chat
  Query: cursor
  Response: Paginated<AvaliacaoChatMensagem>
  AvaliacaoChatMensagem: id, papel: "usuario"|"assistente", conteudo, criado_em

--- Fluxo ALUNO (avaliação) ---

GET /aluno/avaliacoes/disponiveis
  Perfis: A
  Response: lista de itens com id, titulo, pasta_nome, materia_nome, prazo_utc, status_efetivo

GET /aluno/avaliacoes/{avaliacao_id}
  Perfis: A
  Response: AvaliacaoAlunoView (enunciados + alternativas SEM indicar gabarito; texto aberto)
  Erros: 404 se rascunho ou fora do escopo; 403 se encerrada/prazo expirado para NOVA tentativa
    (ainda permitir GET se já existe submissão — política a definir)

POST /aluno/avaliacoes/{avaliacao_id}/submissoes
  Perfis: A
  Response 201: SubmissaoResponse (id, status: "rascunho", iniciada_em)
  RN: avaliacao publicada; dentro do prazo; no máximo uma submissão por (avaliacao, aluno);
    se já existe, retornar 409 com id existente ou reutilizar política idempotente.

PATCH /aluno/submissoes/{submissao_id}
  Body: { "respostas": [ RespostaQuestaoInput, ... ] }  // parcial
  RN: só dono; avaliacao ainda aceita edição (não enviada; prazo válido; não encerrada)
  Response 200: SubmissaoResponse

POST /aluno/submissoes/{submissao_id}/enviar
  Body opcional: comentario_final str
  RN:
    - valida prazo UTC no servidor (RF-009)
    - status avaliacao encerrada ou prazo ultrapassado → 403
    - calcula nota objetiva para MCQ; texto aberto: pontos nulos ou 0 até correção manual
    - enfileira job Relatorio_IA (RF-014)
  Response 200: SubmissaoResponse (status: "enviada", nota_decimal, enviada_em)

3.8 DASHBOARD (OBSERVABILIDADE)
--------------------------------------------------------------------------------

GET /dashboard/resumo
  Query:
    escopo: geral|turma|aluno
    turma_id: UUID (obrigatório se escopo=turma)
    aluno_id: UUID (obrigatório se escopo=aluno)
    data_inicio: date (ISO)
    data_fim: date (ISO)
  Perfis: P (turmas sob gestão); R (alunos dependentes)
  Response 200 (DashboardResumoResponse):
    filtros_aplicados: object
    cartoes:
      total_alunos: int
      media_geral: float
      pendencias_correcao: int  // submissões com questões abertas sem nota final
      taxa_aprovacao: float     // percentual 0–100, regra institucional (ex.: nota >= 6)
    serie_temporal: [
      { periodo: str, media: float, por_turma: { turma_id: float, ... } opcional }
    ]
    disciplinas: [
      { disciplina_id, disciplina_nome, nota_media: float, meta: float }
    ]
    insights: [
      { tipo: str, titulo: str, descricao: str, severidade: "info"|"warning"|"success" }

5. REGRAS DE NEGÓCIO DETALHADAS (CHECKLIST)
================================================================================

5.1 Ciclo de vida da avaliação
- rascunho: visível apenas professor; alunos não listam; edição de questões permitida.
- publicada: visível aos alunos da turma escopada; aceita submissão até prazo_utc ou até
  encerramento manual.
- encerrada: nenhuma nova submissão nem alteração de respostas; transição irreversível MVP.
- publicada → encerrada: professor; automática opcional quando prazo expira (job).

5.2 Edição de conteúdo (paridade UI modulo-avaliacoes)
- modo edição na UI só quando status == rascunho; backend deve retornar 403 em mutações de
  questões se status != rascunho.

5.3 Prazo
- Comparar instante atual UTC com prazo_utc no servidor em toda submissão/envio.
- Encerramento manual prevalece sobre prazo (avaliacao encerrada bloqueia mesmo dentro do prazo).

5.4 Unicidade de submissão
- Unique (avaliacao_id, aluno_id) na tabela submissao.
- Resposta por questão: unique (submissao_id, questao_id).

5.5 Contadores em pasta de avaliações
- alunos_responderam: distinct alunos com submissao status enviada (ou corrigida*) na pasta
  (agregando todas avaliações da pasta conforme regra de produto — definir: por pasta
  significa união de avaliações na pasta; contar aluno que respondeu ao menos uma).
- alunos_total: alunos matriculados na turma escopada da pasta/avaliação.
- avaliacoes_concluidas / avaliacoes_total: contagem por estados das avaliações filhas.
- status_resumo: string derivada (ex.: "1 rascunho · 1 publicada · 1 encerrada").
- Implementação: view materializada ou job de reconciliação; triggers possíveis mas documentar.

5.6 Comunicados
- Validação de criação: titulo não vazio; corpo não vazio OU imagens_urls não vazio.
- Destinatários: pelo menos um.
- Rascunho: visível apenas ao autor na listagem padrão até publicar.

5.7 RBAC
- Middleware de perfil em cada rota; testes de integração para 403 em mutações indevidas.

5.8 IA
- Chat de elaboração: apenas professor; não expor chaves de LLM ao cliente.
- Relatório IA: versão incrementada a cada regeneração; armazenar id_job_auditoria se exigido.

5.9 Privacidade e escopo
- Responsável só acessa dados de alunos vinculados em aluno_responsavel.
- Professor só turmas/instituição associadas (política MVP: professor titular).

11. APÊNDICE B — CÓDIGOS HTTP POR OPERAÇÃO (REFERÊNCIA RÁPIDA)

- 200 OK: GET/PATCH bem-sucedido com corpo
- 201 Created: POST criação
- 202 Accepted: job assíncrono (chat IA)
- 204 No Content: DELETE ou logout
- 400 Bad Request: regra de negócio violada (envelope code)
- 401 Unauthorized: token ausente/inválido
- 403 Forbidden: perfil ou estado (ex.: edição de avaliação não rascunho)
- 404 Not Found: id inexistente ou fora do tenant
- 409 Conflict: unicidade submissão, transição de estado inválida
- 422 Unprocessable Entity: validação Pydantic (campos)
- 429 Too Many Requests: rate limit login/search
- 500 Internal Server Error: falha não tratada
